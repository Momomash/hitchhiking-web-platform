## Заход и постановка темы

- Название (рабочее): **«Автостопом по веб-платформе»** / вариант в этом духе.
- Описание по сути:
    - Когда‑то веб казался простым: странички, ссылки, картинки.
    - Сейчас за окном браузера — огромная экосистема стандартов, API, тестов, соглашений и комитетов.
    - Доклад — «прогулка» по этому миру с ностальгией и фокусом на многослойности платформы.

---

## 2. Ранний веб и ностальгия

- Первый браузер **WorldWideWeb** и изначальная идея веба как гипертекстовой системы для обмена знаниями.
- «Невинный» веб:
    - Статичные HTML‑страницы.
    - Документы, ссылки, немного картинок.
- Ностальгический блок:
    - **Million Dollar Homepage**, стенка из пикселей.
    - Старые сайты типа Space Jam, Flash‑эра, табличная вёрстка.
    - Цель: зацепить эмоциями и показать, как «наивный» веб выглядел.

---

## 3. Скачки во фронтенде и вебе (смены парадигм)

Показать, что фронтенд/веб развиваются скачкообразно и быстрее, чем бэкенд.

- **Скачок 1: статичный HTML → динамический веб**
    - От полных перезагрузок к AJAX, Gmail, Google Maps.
- **Скачок 2: страницы → SPA и фреймворки**
    - Angular / React / Vue / Svelte, клиент как UI‑машина, сервер как API.
- **Скачок 3: десктопный веб → mobile‑first**
    - Адаптивность, touch, производительность, сеть.
- **Скачок 4: веб → PWA**
    - Offline‑first, установка как приложение, push, Service Workers.
- **Скачок 5: классический клиент/сервер → RSC/современные гибриды**
    - Часть рендеринга на сервере, часть на клиенте, новый слой сложности.
- Вставка‑контраст:
    - Бэкенд эволюционирует постепенно (HTTP → REST → GraphQL).
    - Фронтенд и браузер — серия резких смен модели.

## 4. Современный Web Platform и Web API

- Показать, что браузер — не «рендерилка HTML», а платформа:
    - Примеры современных Web API (WebGPU, WebAssembly, WebRTC, Web Bluetooth и т.п. — достаточно нескольких ярких).
    - Браузер как мини‑ОС: доступ к железу, офлайн, пуши, фоновые процессы.
- Связка с предыдущим блоком:
    - Эти API — результат той самой эволюции и давления со стороны приложений/фреймворков.

    - **W3C**, **WHATWG**, **TC39**, **WICG**, OpenJS Foundationи др.
    - Браузерные вендоры (Chrome, Firefox, Safari, Edge) как реальные implementers.
    - web APIs - кто заказчик и исполнитель
- Подчеркнуть смену взглядов и философий:
    - История расхождения W3C vs WHATWG (живой стандарт vs снапшоты).
    - Инкубация идей в WICG, движение в формальные стандарты.
- Фронтенд‑экосистема сверху:
    - React, Next.js, крупные платформы (например, Vercel) как новые центры влияния.
    - Сообщество, которое реагирует на изменения и иногда конфликтует с вендорами.

## 6. Interop и Web Platform Tests

- Проблема: историческая несовместимость браузеров, «браузерные войны» и боль разработчиков.
- Современный ответ:
    - **Web Platform Tests** как общая база тестов для всех браузеров.
    - **Interop** как совместная инициатива выравнивания поведения и приоритизации областей (CSS, forms, scrolling и т.д.).
- Месседж:
    - Мы уже стандартизируем не только текст спецификаций, но и фактическое поведение.
    - Веб становится более предсказуемым, но цена — огромная координация.

## 7. Веб как политико‑правовая конструкция (аккуратный вброс)

- Коротко, без ухода в юризм:
    - Патентная политика W3C, стремление к royalty‑free стандартам.
    - Идея: веб должен быть не только технически открытым, но и **юридически свободным** для реализации.
- Это усиливает мысль: сложность веба — не только в коде, но и в договорённостях.

## 9. Финальный месседж

- Веб давно перестал быть «просто сайтами».
- Браузер — одна из самых сложных и самых согласованных систем, с которыми мы работаем каждый день.
- Каждый новый слой (от ностальгических сайтов до A2UI) добавляет сложность, но остаётся в том же окне — вкладке браузера.
- Если вы в какой-то момент поймаете себя на рефлексии типа “мне скучно“, вспомните с какой ахренительно сложной системой вы работаете :)
